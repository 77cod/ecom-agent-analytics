### 5.1.5 PostgreSQL 日志


## 5.2 后端调试

### 5.2.1 在 VS Code 中对 FastAPI 打断点

**步骤：**

1. 在 VS Code 左侧点击"运行和调试"图标（或按 `Ctrl+Shift+D`）
2. 选择之前创建的 `FastAPI Backend` 配置
3. 在你关心的代码行号左侧点击，出现小红点即为断点
4. 按 `F5` 启动调试
5. 通过浏览器或 `/docs` 发送请求，代码执行到断点时会自动暂停

**推荐断点位置：**

```
D:/industry_information_assistant/backend/app/service/deep_research_v2/graph.py
  第 233 行: async def _plan_node()      -- 规划阶段入口
  第 242 行: async def _research_node()  -- 数据查询入口
  第 250 行: async def _analyze_node()   -- 分析阶段入口
  第 261 行: async def _write_node()     -- 报告撰写入口
  第 269 行: async def _review_node()    -- 审核入口
  第 405 行: async def run_agent_with_streaming() -- Agent 消息流
```

**条件断点：** 右键断点 -> "编辑断点" -> 输入条件，例如 `state["phase"] == "analyzing"`，只在分析阶段触发。

### 5.2.2 如何调试 async 代码

本项目的 DeepResearch Graph 大量使用 `async/await`，调试 async 代码需要注意：

1. **进入 async 函数内部**：按 `F11`（Step Into）会进入 `await` 调用的内部实现
2. **查看协程状态**：在 VS Code 的 "调用堆栈" 面板可以看到 async 调用链
3. **asyncio.Queue 消息跟踪**：在 `run_agent_with_streaming` 函数（`graph.py` 第 405 行）设置断点，可以观察每条 SSE 消息从 Agent 产出到被消费的完整链路

4. **调试技巧：在 Agent 的 `process` 方法入口设置断点**

```python
# D:/industry_information_assistant/backend/app/service/deep_research_v2/agents/base.py 第 44 行
@abstractmethod
async def process(self, state: ResearchState) -> ResearchState:
    pass  # 所有 Agent 都继承这个方法，可以在这里设断点
```

但实际上每个 Agent 都有自己的 `process` 实现。推荐在 `DataQueryer.process()` 和 `CodeWizard.process()` 中设断点，这两个 Agent 最容易出问题。

### 5.2.3 在 VS Code 中观察 `ResearchState`

`ResearchState` 是一个 `TypedDict`，在调试时可以通过 VS Code 的 "监视" 面板直接查看：

- `state["query"]`：用户的原始研究问题
- `state["phase"]`：当前执行阶段（`"init"` / `"researching"` / `"analyzing"` / `"writing"` / `"reviewing"` / `"completed"`）
- `state["facts"]`：收集到的事实列表
- `state["data_points"]`：提取的结构化数据点
- `state["charts"]`：生成的图表列表
- `state["final_report"]`：最终报告文本
- `state["outline"]`：研究大纲
- `state["iteration"]`：当前迭代次数
- `state["messages"]`：SSE 消息队列

**在 VS Code 调试控制台中打印状态：**

```
import json
print(json.dumps(state["facts"][:3], ensure_ascii=False, indent=2))
print(state["final_report"][:200])
```

**在每个 Phase 之间观察状态变化**

在 `graph.py` 的 `_run_simplified` 方法中（第 382 行），每个 Phase 切换时都有明确的 `yield` 点。在每个 `yield {"type": "phase", ...}` 之后设置断点，可以逐阶段观察状态的累积变化。

### 5.3.4 Valtio 状态检查

Valtio 的 `proxy` 状态可以通过以下方式检查：

```typescript
// 在浏览器 Console 中直接访问
// 注意：Valtio proxy 对象需要通过 snapshot 获取快照
import { snapshot } from 'valtio'

// 查看当前行业状态
console.log(snapshot(industryState))

// 查看聊天列表
// 如果 chat 是 proxy，可以这样看：
console.log(snapshot(chat))
```

**Valtio 状态不更新的常见原因：**

1. 直接修改了 proxy 对象中嵌套对象的属性（应该用整体替换）
2. 在 `useSnapshot` 之外读取状态（只能用 `useSnapshot` 或 `snapshot` 读取）
3. 错误的比较逻辑导致 React 跳过渲染

```typescript
// 错误：Valtio 不会追踪这种修改
chat.list.push(newItem)  // 这样可以，但 push 是允许的操作

// 正确：Valtio 支持对 proxy 对象的直接修改
chat.list = [...chat.list, newItem]
```

## 5.5 常见报错及解决方案

以下是本项目开发和使用过程中最常见的 10+ 个错误及其解决方案。

### 5.5.1 ModuleNotFoundError: No module named 'backend'

**错误信息：**

```
ModuleNotFoundError: No module named 'backend'
```

**原因：** Python 的 `sys.path` 中不包含项目的 `backend` 目录的父目录。

**解决方案：**

1. **在 VS Code 中设置 PYTHONPATH：**

```json
// .vscode/settings.json
{
  "python.envFile": "${workspaceFolder}/backend/.env",
  "terminal.integrated.env.windows": {
    "PYTHONPATH": "${workspaceFolder}/backend"
  },
}
```

2. **在脚本入口添加 sys.path 设置：**

```python
import sys
import os
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
```

参考 `test_ecommerce_research.py` 第 23 行和第 52 行的做法：

```python
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
```

3. **运行测试脚本时指定工作目录：**

```bash
cd D:/industry_information_assistant/backend
python -m app.scripts.test_ecommerce_research
```

### 5.5.2 ImportError: attempted relative import

**错误信息：**

```
ImportError: attempted relative import with no known parent package
ImportError: attempted relative import beyond top-level package
```

**原因：** 在直接运行脚本时使用了相对导入（`from .xxx import yyy`），而不是模块导入。

**解决方案：**

本项目中的 `graph.py`、`ecommerce_mock_data.py` 等文件已经通过 try/except 做了兼容性处理：

```python
# graph.py 第 17-27 行
try:
    from router.research_router import is_research_cancelled, clear_cancel_flag
except ImportError:
    try:
        from app.router.research_router import is_research_cancelled, clear_cancel_flag
    except ImportError:
        # 兼容直接运行脚本的情况
        def is_research_cancelled(session_id: str) -> bool:
            return False
```

**通用修复方法：**

- 始终使用 `python -m` 运行脚本而不是 `python script.py`
- 将相对导入（`.xxx`）改为绝对导入（`from app.xxx import yyy`）

### 5.5.3 Connection refused to PostgreSQL / Redis

**错误信息：**

```
sqlalchemy.exc.OperationalError: (psycopg2.OperationalError) connection to server at "localhost" (127.0.0.1), port 5432 failed: Connection refused
redis.exceptions.ConnectionError: Error 10061 connecting to localhost:6379
```

**原因：**

- Docker 容器没有启动
- 端口被其他服务占用
- Docker Compose 中的网络配置问题

**解决方案：**

```bash
# 1. 确认容器是否运行
docker-compose ps

# 2. 如果没有运行，启动它们
docker-compose up -d postgres redis

# 3. 如果端口冲突，查看谁占用了端口
netstat -ano | findstr :5432
netstat -ano | findstr :6379

# 4. 检查容器健康状态
docker inspect industry_postgres | grep -i health
docker inspect industry_redis | grep -i health

# 5. 如果数据库初始化失败，重新创建
docker-compose down -v
docker-compose up -d postgres
```

**注意：** PostgreSQL 和 Redis 的连接配置在 `D:/industry_information_assistant/backend/app/core/database.py` 和 `redis_client.py` 中，默认连接 `localhost`。

### 5.5.5 JSON 解析错误（Agent 响应）

**错误信息：**

```
JSON parse error, could not extract valid JSON
json.decoder.JSONDecodeError: Expecting value: line 1 column 1
```

**原因：** LLM 返回的内容不是合法的 JSON，常见情况：
- LLM 在 JSON 前后加了额外文字
- JSON 中包含非法转义字符（如 `\#`、`\[`）
- JSON 被包裹在 markdown 代码块中（` ```json ... ``` `）

### 5.5.6 前端 CORS 错误

**错误信息（浏览器 Console）：**

```
Access to fetch at 'http://localhost:8000/research/stream' from origin 'http://localhost:5173'
has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present
```

**原因：** 前端（Vite，端口 5173）请求后端（FastAPI，端口 8000）时被浏览器的同源策略拦截。

**解决方案：**

在 `app_main.py` 或路由中配置 CORS 中间件：

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173", "http://localhost:3000"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

同时检查前端的 Vite 代理配置（`vite.config.ts`）：

```typescript
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:8000',
        changeOrigin: true,
      }
    }
  }
})
```

### 5.5.9 Docker 容器启动失败

**常见错误场景：**

**场景 1：端口已被占用**

```bash
Error starting userland proxy: Bind for 0.0.0.0:5432 failed: port is already allocated
```

解决：`netstat -ano | findstr :5432` 找到占用进程，终止或修改 docker-compose.yml 中的端口映射。

**场景 2：磁盘空间不足**

```bash
ERROR: cannot create temporary directory
```

解决：`docker system prune -a` 清理无用的容器、镜像、卷。

**场景 3：PostgreSQL 数据目录损坏**

```bash
FATAL: database files are incompatible with server
```

解决：`docker-compose down -v && docker-compose up -d postgres`（注意 `-v` 会删除数据卷）。

**场景 4：etcd 健康检查失败导致 Milvus 无法启动**

```bash
industry_milvus depends on etcd condition: service_healthy
```

解决：给 etcd 更多启动时间，或者单独重启 etcd：
```bash
docker-compose up -d etcd
# 等 etcd 变成 healthy
docker-compose up -d milvus
```

### 5.5.10 Valtio 状态不更新

**现象：** 修改了 `industryState.currentIndustryId`，但页面没有变化。

**原因分析：**

Valtio 的 `useSnapshot` hook 只对 `proxy` 对象的属性变化做响应。如果使用方式不对，React 不会重新渲染。

**正确用法（`industry.ts` 中的实现）：**

```typescript
// 创建 proxy 状态
export const industryState = proxy<IndustryState>({
  currentIndustryId: getStoredIndustryId(),
  industries: INDUSTRY_CONFIGS,
})

// 在组件中读取
import { useSnapshot } from 'valtio'
const { currentIndustryId } = useSnapshot(industryState)
```

**错误用法 1：** 在 hook 之外读取

```typescript
// 错误：这不会触发重新渲染
const id = industryState.currentIndustryId
```

**错误用法 2：** 解构后修改不会触发更新（需要用整体赋值）

```typescript
// 错误：解构后赋值不触发更新
const state = industryState
state.currentIndustryId = 'new'  // 这样是可以的，直接用 proxy 对象

// 正确：直接修改 proxy 或使用 setter 函数
setCurrentIndustry('beauty')  // 调用预先定义的 setter
```

**错误用法 3：** 忘记 `useSnapshot`

```typescript
// 错误：不会触发重新渲染
const industry = getCurrentIndustry()

// 正确：使用 useSnapshot
const state = useSnapshot(industryState)
```

### 5.5.12 async 调用在非 async 上下文中报错

**错误信息：**

```
RuntimeError: This event loop is already running
RuntimeWarning: coroutine was never awaited
```

**原因：** 在同步代码中调用了 async 函数但没有 `await`，或者在已经有运行中事件循环的环境中又创建了新的事件循环。

**解决方案：**

```python
# 错误：
result = agent.process(state)  # 缺少 await

# 正确：使用 asyncio.run() 或 await
result = await agent.process(state)  # 在 async 函数中
# 或
result = asyncio.run(agent.process(state))  # 在同步代码中
```

## 常见坑

### 坑 1：PYTHONPATH 不一致导致 import 混乱

**现象：** 同一个 import 语句在 VS Code 中可以正常执行，但在终端 `python -m` 时报 `ModuleNotFoundError`。

**原因：** VS Code 的 `PYTHONPATH` 和终端的环境变量不同。VS Code 的 Python 插件会自动把 workspace 根目录加入 `PYTHONPATH`，而终端不会。

**解决方法：** 在项目根目录创建 `.env` 文件统一配置：

```env
PYTHONPATH=backend
```

并在 `launch.json` 中通过 `"envFile"` 引用。

### 坑 2：Windows 路径分隔符导致的跨平台问题

**现象：** 在 Windows 上开发，部署到 Linux 时 `os.path` 相关操作出错。

**解决方法：** 始终使用 `os.path.join()` 和 `pathlib.Path`，不要硬编码 `\` 或 `/`。

```python
# 不好
path = "app/service/" + module_name + ".py"

# 好
from pathlib import Path
path = Path("app") / "service" / f"{module_name}.py"
```

### 坑 3：SSE 流在开发环境下代理失效

**现象：** Vite 的代理配置在 SSE 连接下可能不正常工作。

**原因：** Vite 的 proxy 默认会缓冲响应，这破坏了 SSE 的实时流特性。

**解决方法：** 在 `vite.config.ts` 的代理配置中关闭事件源缓冲：

```typescript
server: {
  proxy: {
    '/api': {
      target: 'http://localhost:8000',
      changeOrigin: true,
      configure: (proxy) => {
        proxy.on('proxyReq', (proxyReq) => {
          // 对于 SSE 请求，确保不使用缓冲
        })
      }
    }
  }
}
```

或者在开发时直接用完整 URL 请求后端，绕过代理：
```typescript
const API_BASE = 'http://localhost:8000'
fetch(`${API_BASE}/research/stream`, { ... })
```

### 坑 4：DeepScout 已被废弃但仍出现在导入中

**现象：** `graph.py` 第 38 行导入了 `DeepScout`，`__init__.py` 第 9 行也导出了它，但在电商版本中 `DataQueryer` 替代了它的功能。

**说明：** `DeepScout` 是旧版用于搜索互联网的 Agent，在电商版中已被 `DataQueryer` 替代。这是第 6 章需要清理的"死代码"之一。

### 坑 5：validate=False 导致 Pydantic 校验被跳过

**现象：** 请求体中的某些字段类型不对，但没有报错，数据静默丢失。

**原因：** 如果在某处使用了 `model_validate(data)` 或者有 `validate: false` 配置，Pydantic 可能跳过了类型校验。

**解决方法：** 在 Router 层的 Pydantic Model 中不要使用 `ConfigDict(validate_assignment=False)`。
