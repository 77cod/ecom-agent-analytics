## 7.3 从 0 到 1 搭建同类项目的步骤

基于本项目的经验，将搭建一个多 Agent 分析平台的通用步骤总结如下：

### 第一阶段：需求定义与架构设计（1-2 天）

```
1. 明确领域和用户
   - 本项目：电商运营人员，需要分析品牌经营数据
   - 你可以：金融分析师、医疗研究员、教育管理者...

2. 定义 Agent 角色
   - 问自己：一个完整分析流程需要哪些专家？
   - 本项目：规划师 + 数据查询员 + 数据分析师 + 代码工程师 + 写手 + 审核员
   - 通用模式：分析师 + 研究员 + 审核员 至少 3 个

3. 设计状态结构
   - 问自己：Agent 之间需要共享什么信息？
   - 本项目：ResearchState (query, facts, data_points, charts, final_report...)
   - 通用字段：query, intermediate_results, final_output, iteration_count
```

### 第二阶段：Mock 数据先行（1-2 天）

```
1. 构造有"故事性"的数据
   - 不要随机生成无意义数据
   - 埋入"问题"和"机会点"，让 AI 有东西可分析
   - 本项目示例：竞品价格更低且正在增长、差评集中在特定问题

2. 数据结构设计
   - 确保包含至少 3 个维度的数据：
     a. 时间维度（月度趋势）
     b. 分类维度（商品/品类）
     c. 对比维度（自家 vs 竞品）

3. 用 Python 字典模拟
   - 简单直接，无需数据库
   - 可以随时修改数据结构
```

### 第三阶段：搭建 Agent 框架（2-3 天）

```
1. 创建 Agent 基类
   - 定义通用接口：process(state) -> state
   - 封装 LLM 调用方法
   - 实现日志记录

2. 实现各个 Agent
   - 从最简单的开始（如数据查询）
   - 逐个实现并单元测试
   - 每个 Agent 只做一件事

3. 编排工作流
   - 确定执行顺序（线性 or DAG）
   - 实现条件分支（审核不通过 -> 修订 -> 再审核）
   - 添加取消和检查点机制
```

### 第四阶段：搭建前后端接口（1-2 天）

```
1. 后端 API（FastAPI）
   - POST /research/stream：发起研究（SSE 流式）
   - POST /research/cancel/{id}：取消研究
   - GET /research/checkpoint/{id}：获取检查点

2. 前端页面（React）
   - 输入区：问题输入 + 行业选择
   - 进程区：Agent 执行步骤展示
   - 结果区：分析报告 + ECharts 图表

3. SSE 实时推送
   - 后端：asyncio.Queue 收集 Agent 消息
   - 前端：fetch + ReadableStream 接收 SSE
```

### 第五阶段：优化与扩展（持续）

```
1. 性能优化
   - LLM 调用缓存 (Redis)
   - 前端虚拟滚动 (react-window)
   - React.memo 避免不必要渲染

2. 功能扩展
   - 添加新的分析维度/Agent
   - 支持新的数据源
   - 导出 PDF/Excel 报告

3. 工程化
   - Git 规范化
   - CI/CD
   - 单元测试 + 端到端测试
```

### 时间估算

| 阶段 | 最少时间 | 推荐时间 | 产出 |
|------|---------|---------|------|
| 需求 + 架构 | 4 小时 | 2 天 | 架构图 + 接口定义 |
| Mock 数据 | 3 小时 | 1 天 | Mock 数据集 |
| Agent 框架 | 8 小时 | 3 天 | 可工作的 Agent 链 |
| 前后端接口 | 6 小时 | 2 天 | 可用的 Web 应用 |
| 优化扩展 | 4 小时/项 | 持续 | 更完善的系统 |
| **总计** | **~25 小时** | **1-2 周** | **完整 MVP** |

## 7.4 结业实战任务

### 任务 1（简单）：添加新行业并生成完整运营分析报告

**目标：** 为平台添加第 5 个行业"母婴用品"，并端到端测试整个分析流程。

**具体要求：**

1. 按照第 6 章 6.4 节的步骤，完成后端和前端共 5 个文件的修改
2. Mock 数据至少包含：5 个自家商品 + 3 个竞品商品、6 个月销售趋势、8 条以上用户评价（含好评差评）、3 个达人带货记录、竞品价格对比
3. 运行 `test_ecommerce_research.py`，确认流程正常
4. 在前端选择"母婴用品"行业，输入问题"纸尿裤品牌'宝贝舒'的 8 月经营分析报告"，获得包含图表和文字的分析报告

**验收标准：**

- [ ] 前端行业选择器中出现"母婴用品"选项
- [ ] 后端能正确加载 `ecommerce_mock_data_baby.py` 的数据
- [ ] 报告中提到"宝贝舒"和"花之妙"品牌名称
- [ ] 至少生成 2 种不同类型的图表
- [ ] 报告长度 > 1000 字

### 任务 2（中等）：为仪表盘添加雷达图

**目标：** 在现有图表类型的基础上，增加雷达图支持，用于竞品多维对比分析。

**具体要求：**

1. 扩展前端的 `ChartType` 类型定义，增加 `'radar'`
2. 在 `Chart` 组件中添加雷达图的渲染逻辑（利用 ECharts 的 `radar` 类型）
3. 后端 `chart_generator.py` 中添加 `generate_radar_chart()` 方法
4. 修改 `CodeWizard` 的 prompt，使其在合适的场景下生成雷达图（如竞品对比分析）
5. 新增的数据必须通过雷达图正确展示

**验收标准：**

- [ ] 前端支持 `type: 'radar'` 的图表配置
- [ ] 雷达图能正确显示多维数据（至少 4 个指标）
- [ ] 支持多系列数据（至少 2-3 个对比项）
- [ ] 图表交互正常（hover 显示数据、图例切换）

### 任务 3（困难）：将 Mock 数据替换为真实数据库

**目标：** 将任意一个行业的数据从 Python 文件迁移到 PostgreSQL，实现真实的数据持久化。

**具体要求：**

1. 创建 SQLAlchemy ORM 模型（至少 3 张表：products、sales、reviews）
2. 编写数据迁移脚本，将现有 mock 数据导入数据库
3. 创建 `EcommerceRepository` 数据访问层
4. 修改 `DataQueryer`，支持从数据库查询数据（通过环境变量切换）
5. 添加一个管理接口：`POST /admin/import-data` 用于导入 Excel/CSV 的电商数据
6. 系统能同时在 mock 和 database 模式下运行，由 `.env` 中的 `ECOMMERCE_DATA_SOURCE` 控制

**验收标准：**

- [ ] 3 张表通过 SQLAlchemy 正确创建
- [ ] 迁移脚本成功将 mock 数据导入数据库
- [ ] 设置 `ECOMMERCE_DATA_SOURCE=database` 后，研究流程使用数据库数据
- [ ] 数据库查询性能 < 200ms（通过 Redis 缓存或 SQL 优化）
- [ ] 错误处理完善（数据库连接失败时给出友好提示）

## 7.5 进阶学习路线

完成本项目后，建议按以下路线继续深入学习：

### 方向 1：Agent 与 Prompt Engineering

```
现在: 7 个预定义 Agent，固定 prompt

下一步:
  - 动态 Prompt 生成（根据用户问题和行业调整 system prompt）
  - Multi-Agent 辩论机制（两个 Agent 辩论不同观点，产生更全面的分析）
  - Tool Use / Function Calling（Agent 主动调用外部工具）
  - 学习 LangChain / LangGraph 高级特性

推荐资源:
  - LangGraph 官方文档：条件边、并行执行、流式输出
  - Anthropic 的 "Building Effective Agents" 文章
  - OpenAI Function Calling 文档
```

### 方向 2：数据分析与可视化

```
现在: CodeWizard 生成 ECharts 配置 JSON

下一步:
  - 实时数据看板（WebSocket 推送更新）
  - 多维数据下钻（点击图表进入详情）
  - 导出 PDF 报告（包含图表和文字）
  - 数据异常检测（自动告警：销量突然下降 > 30%）
  - 时间序列预测（LLM + 传统模型混合预测）

推荐资源:
  - ECharts 官方示例集（echarts.apache.org/examples）
  - Apache Superset（开源 BI 工具，可参考其设计）
  - Prophet / statsmodels 时间序列预测
```

### 方向 3：工程化与 DevOps


## 7.6 常见面试题

以下是关于本项目的 5 个高频面试题及参考回答：

### 面试题 1：请介绍一下你做过的这个电商运营分析平台

**参考回答：**

我参与开发了一个基于多 Agent 协作的智能电商运营分析平台。用户选择一个行业（如服装鞋包、美妆个护），输入分析问题（如"防晒衣品牌 8 月经营分析"），系统会自动调度 6 个 AI Agent 协作完成分析：

- **ChiefArchitect** 规划分析维度
- **DataQueryer** 查询电商经营数据
- **DataAnalyst** 提取结构化数据
- **CodeWizard** 生成 ECharts 可视化图表
- **LeadWriter** 撰写运营分析报告
- **CriticMaster** 对抗式审核报告质量

技术栈用的是 FastAPI + React + PostgreSQL + Redis，Agent 之间通过共享的 ResearchState 传递信息，前端通过 SSE 实时展示分析进度和中间结果。项目支持 4 个行业、12 种以上的图表类型、研究断点续传和取消功能。

**面试官可能会追问：**
- "多 Agent 之间如何协调？"
- "遇到 Agent 输出不稳定怎么办？"
- "如何处理 LLM 调用失败的情况？"

### 面试题 2：为什么选择 LangGraph（或手写状态机）而不是 LangChain？

**参考回答：**

在我们的项目中，我们实际上写了自己的状态机实现，而不是完全依赖 LangGraph。主要考虑：

1. **实时流式输出**：LangGraph 的 `astream` 是批量输出的，无法实现真正的逐条实时推送。我们使用了 `asyncio.Queue` 作为 Agent 和 Router 之间的桥梁，每个 Agent 产出消息后立即推送给前端。

2. **可控性**：手写状态机让我们对每个 Phase 的执行有完全的控制权，包括取消检查、检查点保存时机、错误恢复策略等。

3. **依赖复杂度**：LangGraph 目前仍在快速迭代，API 不稳定。我们通过 try/except 兼容了有无 LangGraph 的场景，默认使用手写版本。

不过对于更复杂的条件分支和并行执行场景，LangGraph 的声明式 API 确实更有优势，我们在非实时的批处理场景中会考虑使用它。

### 面试题 3：如何处理多个 Agent 的输出稳定性问题？

**参考回答：**

这是一个核心挑战。我们采用了以下几个策略：

1. **JSON Mode**：所有 Agent 的 LLM 调用都使用 `response_format: json_object`，确保输出格式可解析。即使 LLM 的输出中存在非法 JSON，我们的 `parse_json_response` 方法也有 4 层 fallback：直接解析 -> 提取 markdown 代码块 -> 提取大括号内容 -> `ast.literal_eval` 兜底。

2. **结构化 Prompt**：每个 Agent 的 system prompt 都明确规定了输出 JSON 的 schema，使用 Zod/JSON Schema 风格的描述。

3. **CriticMaster 审核**：在所有 Agent 完成工作后，CriticMaster 会对报告进行多维度评分（数据准确性、逻辑连贯性、可操作性等）。低于质量阈值的报告会被打回修订。

4. **迭代次数限制**：设置 `max_iterations`（默认为 3），避免无限循环修订。

5. **降级策略**：如果 JSON 解析完全失败，返回空字典，流程继续，由后续 Agent 或人工介入处理。

### 面试题 4：SSE 流式推送在项目中是如何实现的？

**参考回答：**

SSE（Server-Sent Events）是本项目的核心技术之一，用于将多 Agent 的分析进度实时推送到前端。

**后端实现：**

1. FastAPI 的 `/research/stream` 端点返回 `StreamingResponse`，`media_type="text/event-stream"`
2. `DeepResearchGraph._run_simplified()` 中使用 `asyncio.Queue` 收集各 Agent 产出的消息
3. 每个 Agent 调用 `add_message()` 时，消息被放入 `_message_queue`
4. 图的主循环通过 `run_agent_with_streaming()` 持续从队列取出消息并 `yield`
5. 最终每个事件被格式化为 SSE 格式：`data: {"type":"...", ...}\n\n`

**前端实现：**

1. 使用 `fetch` API 发起请求，获取 `response.body` 作为 `ReadableStream`
2. 通过 `reader.read()` 循环读取 SSE 数据
3. 解析每行 `data:` 前缀的内容为 JSON
4. 根据事件类型更新 React 状态（研究步骤、图表、报告文本）

**关键设计决策：**

- 不使用 WebSocket 而选择 SSE，因为 SSE 更简单，单向推送即可满足需求
- 支持研究取消：前端的 `reader.cancel()` 和后端的 Redis 取消标志配合

### 面试题 5：如果要支持 100 个并发用户，你会怎么优化？

**参考回答：**

我会从以下几个层面进行优化：

**1. LLM 调用层面（最大瓶颈）：**

- 使用异步并发 + 连接池，限制同时进行的 LLM 调用数量
- 对相同/相似的问题使用 Redis 缓存研究结果
- 考虑使用更快的模型（如 `qwen-plus` 替代 `qwen-max`）处理非核心 Agent
- 实现请求排队机制（如 Bull/BullMQ），避免 LLM API 超频

**2. 数据库层面：**

- 为高频查询字段添加数据库索引
- 使用 Redis 缓存行业配置、热门数据等不常变化的数据
- 实现读写分离（PostgreSQL 主从复制）

**3. 前端层面：**

- 研究列表使用虚拟滚动（react-window）
- 图表组件懒加载 + 防抖重渲染
- 使用 Service Worker 缓存静态资源

**4. 部署层面：**

- 后端水平扩展（多实例 + Nginx 负载均衡）
- 数据库连接池优化（`max_connections` 调大，使用 PgBouncer）
- SSE 连接管理（Nginx 配置 `proxy_buffering off`）

**5. 监控层面：**

- Prometheus + Grafana 监控 API 延迟、错误率、LLM 调用次数
- 设置告警规则（API 错误率 > 5%、平均延迟 > 10s）
