## 3.1 分层架构设计

整个项目采用**经典三层架构 + Agent 层**：

```
┌──────────────────────────────────────┐
│          表示层 (Presentation)        │
│  React 页面 → Ant Design 组件         │
│  Valtio 全局状态 → Axios API 请求     │
└────────────────┬─────────────────────┘
                 │ HTTP / SSE
┌────────────────▼─────────────────────┐
│          路由层 (Router)              │
│  11 个 FastAPI Router                 │
│  参数校验 (Pydantic) → 分发到 Service │
└────────────────┬─────────────────────┘
                 │
┌────────────────▼─────────────────────┐
│         业务逻辑层 (Service)          │
│  DeepResearchV2Service (多Agent)      │
│  ChatService (RAG对话)               │
│  KnowledgeService (文档管理)          │
└────────────────┬─────────────────────┘
                 │
┌────────────────▼─────────────────────┐
│          Agent 协作层                 │
│  DeepResearchGraph (工作流编排)       │
│  7 Agents 读写共享 ResearchState      │
│  asyncio.Queue (SSE 实时推送)        │
└────────────────┬─────────────────────┘
                 │
┌────────────────▼─────────────────────┐
│          数据访问层 (Model/ORM)        │
│  SQLAlchemy ORM → PostgreSQL          │
│  Milvus Client → 向量搜索             │
│  Redis → 缓存/研究取消               │
└──────────────────────────────────────┘
```

### 为什么这样分层

1. **表示层独立**：前端只关心展示和交互，不关心数据怎么查的
2. **路由层薄**：只做参数校验和分发，业务逻辑全在 Service
3. **Agent 层解耦**：每个 Agent 独立开发、独立测试，通过共享状态协作
4. **数据层统一**：换数据库只需改 Model 层

## 3.2 核心设计：共享状态

整个 Agent 系统的核心是一个共享的 `ResearchState` 字典：

```python
# state.py
class ResearchState(TypedDict):
    # 基础信息
    query: str                    # 用户问题
    session_id: str               # 会话ID
    industry_id: str              # 行业 (fashion/beauty/digital/food)
    phase: str                    # 当前阶段
    iteration: int                # 当前轮次

    # 规划输出
    outline: List[Dict]           # 分析大纲
    hypotheses: List[Dict]        # 研究假设
    research_questions: List[str] # 待研究问题

    # 知识库
    facts: List[Dict]             # 结构化事实
    data_points: List[Dict]       # 数据点 (GMV, ROI...)

    # 分析输出
    charts: List[Dict]            # 图表配置
    insights: List[str]           # 数据洞察

    # 写作输出
    draft_sections: Dict          # 章节草稿
    final_report: str             # 最终报告

    # 审核
    critic_feedback: List[Dict]   # 审核意见
    quality_score: float          # 质量评分

    # 流式输出
    messages: List[Dict]          # SSE 事件队列
```

### 为什么用共享状态而不是消息传递

```
方案 A：消息传递              方案 B：共享状态（本项目）
Agent A → msg → Agent B      Agent A → 写 state
Agent B → msg → Agent C      Agent B → 读 state → 写 state
Agent C → msg → Agent A      Agent C → 读 state → 写 state

问题：                        优势：
- 消息格式要统一               - 所有 Agent 看到全局信息
- 顺序耦合强                   - 任意 Agent 可查任何数据
- 难以恢复/暂停                - 序列化 state 即可保存检查点
```

## 3.3 工作流编排：DeepResearchGraph

### 状态机设计

整个研究流程是一个有限状态机：

```
                    ┌──────────────┐
                    │     INIT     │  初始状态
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   PLANNING   │  Architect 分析问题
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  RESEARCHING │  DataQueryer 查询数据
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  ANALYZING   │  DataAnalyst + CodeWizard
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   WRITING    │  LeadWriter 写报告
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  REVIEWING   │  CriticMaster 审核
                    └──┬───────┬───┘
               score<6 │       │ score≥6
               ┌───────▼─┐  ┌──▼──────────┐
               │REVISING │  │  COMPLETED   │
               └────┬────┘  └──────────────┘
                    │
                    └──→ 回到 REVIEWING
```

### 代码实现要点

```python
# graph.py
class DeepResearchGraph:
    async def run(self, query, session_id, industry_id):
        # 1. 创建初始状态
        state = create_initial_state(query, session_id, industry_id=industry_id)

        # 2. 按阶段依次执行
        state = await self.architect.process(state)     # PLAN
        state = await self.data_queryer.process(state)  # RESEARCH
        state = await self.data_analyst.process(state)  # ANALYZE (part 1)
        state = await self.wizard.process(state)        # ANALYZE (part 2)
        state = await self.writer.process(state)        # WRITE
        state = await self.critic.process(state)        # REVIEW

        # 3. 审核不通过 → 修订循环
        while state['quality_score'] < 6.0 and state['iteration'] < max:
            state = await self.writer.process(state)    # REVISE
            state = await self.critic.process(state)    # REVIEW again

        return state
```

## 3.4 SSE 流式推送实现

### 什么是 SSE

SSE（Server-Sent Events）是一种服务器向浏览器单向推送数据的技术。相比 WebSocket，SSE 更简单，适合"服务器持续有进度要汇报"的场景。

```
普通 HTTP：  客户端 ──req──→ 服务器 ──res──→ 客户端  (一次)
SSE：        客户端 ──req──→ 服务器 ──data──→ 客户端
                            服务器 ──data──→ 客户端
                            服务器 ──data──→ 客户端  (持续)
                            服务器 ──[DONE]─→ 客户端
```

### SSE 事件类型

| 事件类型 | 触发时机 | 前端效果 |
|----------|----------|----------|
| `research_start` | 研究开始 | 初始化研究步骤 |
| `research_step` | 每个阶段 | 更新时间线进度 |
| `thought` | Agent 思考中 | 显示"思考中…" |
| `outline` | 规划完成 | 显示分析大纲 |
| `fact` | 查到数据 | 搜索结果列表 |
| `chart` | 图表生成 | 渲染 ECharts 图表 |
| `draft` | 章节写完 | 显示报告片段 |
| `critique` | 审核完成 | 显示审核意见 |
| `research_complete` | 研究完成 | 显示完整报告 |
| `error` | 出错 | 显示错误信息 |
| `[DONE]` | 流结束 | 标记完成 |

## 3.5 前端状态管理：Valtio

### 为什么选 Valtio

Valtio 是一个轻量级的状态管理库，基于 JavaScript Proxy 实现响应式：

```typescript
// store/industry.ts
import { proxy } from 'valtio'

export const industryState = proxy({
  currentIndustryId: 'fashion',  // 当前选中行业
  industries: [...],             // 行业列表
})

// 任何组件都可以直接读写
import { useSnapshot } from 'valtio'
const industry = useSnapshot(industryState)
console.log(industry.currentIndustryId)  // 'fashion'

// 修改自动触发重渲染
industryState.currentIndustryId = 'beauty'
```

相比 Redux 的优势：
- 不需要写 reducer/action
- 不需要 dispatch
- 直接赋值就触发更新
- 配合 `useSnapshot` 自动追踪依赖

### 全局状态一览

| Store | 文件 | 存储内容 |
|-------|------|----------|
| `authState` | `store/auth.ts` | 用户登录状态、Token |
| `sessionState` | `store/session.ts` | 聊天会话列表 |

## 3.6 前端 API 层设计

### axios 插件架构

```typescript
// api/request/
createRequest({
  plugins: [
    authPlugin,      // 自动加 Authorization header
    errorPlugin,     // 全局错误弹窗
    loadingPlugin,   // 全局加载动画
    repeatPlugin,    // 取消重复请求
  ]
})
```

每个插件是一个中间件，在请求的不同阶段执行：

```
请求发起 → authPlugin(加token) → loadingPlugin(显示loading)
    → 发送请求
    → 收到响应 → errorPlugin(检查错误) → loadingPlugin(隐藏loading)
    → 返回结果
```
