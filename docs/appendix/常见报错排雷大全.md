## 一、Docker 相关

### 1. 端口冲突

**报错信息：**
```
Error response from daemon: Ports are not available:
port 8000 is already in use
```

**可能原因：**
- 宿主机已有其他进程占用 8000 端口（如上次未关闭的 uvicorn 进程）
- 另一个 Docker 容器绑定了同一端口
- Windows 下 Hyper-V 保留了该端口

**解决方案：**
```bash
# 1. 查找占用端口的进程
netstat -ano | findstr :8000          # Windows
lsof -i :8000                         # Linux/Mac

# 2. 杀死占用进程（Windows）
taskkill /PID <进程ID> /F

# 3. 或修改 docker-compose.yml 中的端口映射
ports:
  - "8001:8000"   # 改用 8001
```

### 2. 内存不足导致容器崩溃

**报错信息：**
```
exited with code 137
```

**可能原因：**
- Docker Desktop 分配的内存限额过小（默认 2GB）
- 同时运行 PostgreSQL + Redis + Milvus + Elasticsearch 等内存密集型服务超出上限
- 某个服务存在内存泄漏

**解决方案：**
```bash
# 1. 查看容器退出状态
docker compose ps -a

# 2. Docker Desktop 设置中增加内存限制到 8GB+
#    Settings -> Resources -> Memory

# 3. 仅启动必要服务
docker compose up -d backend postgres redis

# 4. 查看容器资源消耗
docker stats
```

---

### 3. Volume 权限问题

**报错信息：**
```
Error: EACCES: permission denied, mkdir '/app/data'
```

**可能原因：**
- 宿主机上创建的 volume 目录属于 root，容器内应用用户无写入权限
- Windows 下 WSL 文件权限映射异常

**解决方案：**
```bash
# 1. 修改目录权限（Linux/Mac）
sudo chown -R 1000:1000 ./data

# 2. 删除有问题的 volume 重建
docker compose down -v
docker compose up -d

# 3. 在 Dockerfile 中指定用户
USER 1000:1000
```

```bash
PYTHONPATH=. python app/main.py
```

### 7. asyncio 事件循环冲突

**报错信息：**
```
RuntimeError: This event loop is already running
```

**可能原因：**
- 在已有事件循环的线程中调用 `asyncio.run()`
- Jupyter Notebook 环境中运行异步代码
- Uvicorn 使用了不同的循环策略

**解决方案：**
```python
# 1. 在已有循环中使用 await 代替 asyncio.run()
import asyncio
try:
    loop = asyncio.get_running_loop()
    # 已有循环，使用 await
    result = await async_function()
except RuntimeError:
    # 无运行中的循环，可以安全调用 asyncio.run()
    result = asyncio.run(async_function())

# 2. 在 FastAPI 路由中使用 async def 自动处理
@app.post("/analyze")
async def analyze(request: Request):  # FastAPI 自动管理事件循环
    return await do_analysis()
```

## 三、Node / 前端

### 8. npm ERESOLVE 依赖冲突

**报错信息：**
```
npm ERR! code ERESOLVE
npm ERR! Unable to resolve dependency tree
```

**可能原因：**
- 某些包的版本要求与 peerDependencies 冲突
- npm 7+ 比 npm 6 更严格的依赖解析策略
- lock 文件与 package.json 不同步

**解决方案：**
```bash
# 1. 清理并重装（推荐）
rm -rf node_modules package-lock.json
npm install

# 2. 使用 --legacy-peer-deps 绕过冲突
npm install --legacy-peer-deps

# 3. 指定具体版本安装冲突包
npm install antd@5.12.0 --legacy-peer-deps

# 4. 更新 lock 文件到 v3 格式
npm install --package-lock-only
```

```
Uncaught Error: Cannot find module 'react'
```

```bash
    '@components': '/src/components'

}

# 3. 同时配置 tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

### 10. 前端空白页面

**报错信息：**
```
浏览器显示白屏，控制台无显式错误或仅有 404
```

**可能原因：**
- React 组件渲染过程中抛出未捕获异常（React 18 会吞掉错误导致白屏）
- 路由配置不匹配当前 URL，匹配到空组件

```bash
# src/components/ErrorBoundary.tsx
class ErrorBoundary extends React.Component {
  componentDidCatch(error, info) {
    console.error("Caught by boundary:", error, info);
  }
  render() {
    return this.props.children;
  }
}

# 2. 清除浏览器硬性缓存
# Chrome: F12 -> Network 标签 -> 勾选 Disable cache

# 3. 检查路由通配符
# 确保 Router 中有兜底路由
<Route path="*" element={<NotFound />} />

# 4. 检查浏览器控制台 Network 标签，确认 API 请求是否正常
```

### 11. ECharts 图表不渲染

**报错信息：**
```
Uncaught TypeError: Cannot read properties of undefined (reading 'width')
```

**可能原因：**
- 图表容器 DOM 元素尚未挂载就调用 `echarts.init()`
- 容器宽高为 0（未设置或 `display: none`）
- `setOption()` 时数据格式与图表类型不匹配

**解决方案：**
```tsx
// 1. 使用 useEffect + useRef 确保容器已渲染
import { useEffect, useRef } from 'react';
import * as echarts from 'echarts';

function Chart() {
  const chartRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (!chartRef.current) return;
    const chart = echarts.init(chartRef.current);
    chart.setOption({ /* options */ });

    // 2. 监听窗口 resize 自动调整
    const handleResize = () => chart.resize();
    window.addEventListener('resize', handleResize);

    return () => {
      chart.dispose();
      window.removeEventListener('resize', handleResize);
    };
  }, []);

  // 3. 给容器明确的宽高
  return <div ref={chartRef} style={{ width: '100%', height: 400 }} />;
}
```


**可能原因：**

```bash
curl -X POST http://localhost:8000/api/v1/auth/login \

# 3. 检查 .env 中的 SECRET_KEY 是否稳定

JWT_SECRET_KEY=your-fixed-secret-key-here
```

**报错信息：**
```
HTTP 500 Internal Server Error
{"detail": "Internal Server Error"}
```

**可能原因：**
- 后端代码抛出未捕获异常（数据库连接失败、空指针等）

```bash
docker compose exec backend alembic upgrade head

# 4. 在代码中添加全局异常处理器查看详细错误
# app/main.py
from fastapi import Request
from fastapi.responses import JSONResponse

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    import traceback
    traceback.print_exc()  # 打印完整错误栈
```

### 14. CORS 跨域错误

**报错信息：**
```
Access to XMLHttpRequest at 'http://localhost:8000/api/v1/analysis'
from origin 'http://localhost:5173' has been blocked by CORS policy
```

**可能原因：**
- 后端 CORS 中间件未配置或未允许前端源
- `credentials: 'include'` 模式下 `Access-Control-Allow-Origin` 不能为 `*`
- 预检请求（OPTIONS）被中间件拦截

**解决方案：**
```python
# app/main.py
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",    # Vite 开发服务器
        "http://localhost:3000",    # 备用端口
    ],
    allow_credentials=True,         # 允许携带 Cookie/Token
    allow_methods=["*"],            # 允许所有 HTTP 方法
    allow_headers=["*"],            # 允许所有请求头
)
```

## 六、Agent 相关

### 17. JSON 解析错误

**报错信息：**
```
json.decoder.JSONDecodeError: Expecting value: line 1 column 1 (char 0)
```

**可能原因：**
- LLM 返回的内容不是合法 JSON（可能包含 markdown 代码块包裹）
- prompt 要求输出 JSON 但模型自由发挥了文字
- 网络中断导致返回内容被截断

**解决方案：**
```python
import json
import re

def parse_llm_json(raw_output: str) -> dict:
    # 1. 尝试直接解析
    try:
        return json.loads(raw_output)
    except json.JSONDecodeError:
        pass

    # 2. 提取 markdown 代码块中的 JSON
    match = re.search(r'```(?:json)?\s*([\s\S]*?)```', raw_output)
    if match:
        try:
            return json.loads(match.group(1).strip())
        except json.JSONDecodeError:
            pass

    # 3. 降级处理：记录原始输出并返回空结构
    print(f"[WARN] Cannot parse JSON from LLM output: {raw_output[:200]}...")
    return {"error": "parse_failed", "raw": raw_output}

# 4. 在 system_prompt 中明确格式要求
system_prompt = """
你必须以纯 JSON 格式输出，不要包含 markdown 代码块标记。
示例输出：{"key": "value"}
"""
```

### 19. LangGraph 节点属性错误

**报错信息：**
```
AttributeError: 'AgentState' object has no attribute 'search_results'
```

**可能原因：**
- State TypedDict 中某个字段拼写不一致（`search_result` vs `search_results`）

```python
    return state

# 3. 使用 .get() 安全访问
results = state.get('search_results', [])
```

## 七、前端状态问题

### 20. Valtio 状态不更新

**现象：**
组件使用了 `useSnapshot` 但状态变化后界面不刷新。

**可能原因：**
- 直接赋值替换了 proxy 对象而非修改其属性
- 在异步回调中修改状态而未使用 `subscribeKey`
- 组件未使用 `useSnapshot` 包裹，直接读取了 proxy 对象

**解决方案：**
```tsx
import { proxy, useSnapshot } from 'valtio';

// 1. 正确：修改 proxy 的属性
const store = proxy({ reports: [] });
store.reports.push(newReport);          // 会触发更新
store.reports = [...store.reports, newReport];  // 也会触发更新

// 2. 错误：直接替换掉 proxy 本身
// store = proxy({ reports: newList });   // 不会触发更新！

// 3. 在组件中正确使用
function ReportList() {
  const snap = useSnapshot(store);  // 必须用 useSnapshot
  return <div>{snap.reports.length}</div>;  // 通过 snap 读取
}
```

### 21. React 状态闭包陷阱

**现象：**
`setTimeout` 或 `setInterval` 中读取的状态始终是初始值。

**可能原因：**
- 回调函数捕获了创建时的闭包变量，每次渲染都不会更新引用

**解决方案：**
```tsx
// 错误：count 永远输出 0
useEffect(() => {
  const timer = setInterval(() => {
    console.log(count);  // 永远是初始值
  }, 1000);
  return () => clearInterval(timer);
}, []);

// 正确方案 1：使用 useRef 保存最新值
const countRef = useRef(count);
countRef.current = count;
useEffect(() => {
  const timer = setInterval(() => {
    console.log(countRef.current);  // 始终获取最新值
  }, 1000);
  return () => clearInterval(timer);
}, []);

// 正确方案 2：使用函数式更新
const [count, setCount] = useState(0);
useEffect(() => {
  const timer = setInterval(() => {
    setCount(prev => prev + 1);  // prev 始终是最新值
  }, 1000);
  return () => clearInterval(timer);
}, []);
```
