```python
# line 289-350
async def run(self, query, session_id, resume=False,
              user_id=None, search_web=True, search_local=False,
              industry_id="fashion"):
    # 尝试从检查点恢复
    state = None
    if resume and session_id:
        state = self._load_checkpoint(session_id)

    # 创建初始状态
    if not state:
        state = create_initial_state(
            query, session_id,
            search_web=search_web, search_local=search_local,
            industry_id=industry_id
        )
        yield {"type": "research_start", "query": query}

    # 按阶段依次执行 Agent
    state = await self.architect.process(state)      # ① 规划
    state = await self.data_queryer.process(state)   # ② 查询
    state = await self.data_analyst.process(state)   # ③ 分析(1)
    state = await self.wizard.process(state)         # ③ 分析(2)
    state = await self.writer.process(state)         # ④ 撰写
    state = await self.critic.process(state)         # ⑤ 审核

    # 审核不通过 → 修订循环
    while (state["quality_score"] < 6.0 and
           state["iteration"] < self.max_iterations):
        state = await self.writer.process(state)     # 修订
        state = await self.critic.process(state)     # 再审
        state["iteration"] += 1
```

**逐行讲解：**

- `resume` — 支持从检查点恢复中断的研究
- `yield {"type": "research_start"}` — 用 `yield` 实现流式输出，每次 yield 一个事件给 SSE
- 每个 `await xxx.process(state)` 是同步等待，Agent 之间串行执行
- 审核循环：`quality_score < 6.0` 触发修订，最多 `max_iterations` 轮
- DataAnalyst 和 CodeWizard 在同一个 ANALYZE 阶段先后执行

**设计要点：**

为什么 Agents 串行而非并行？因为每个 Agent 依赖前一个 Agent 的输出：
- Writer 需要 DataAnalyst 提取的指标
- Critic 需要 Writer 写好的报告
- 串行保证了数据依赖关系

**核心代码 — call_llm：**

```python
async def call_llm(self, system_prompt, user_prompt,
                    json_mode=False, temperature=None, max_tokens=None):
    """调用 LLM - 所有 Agent 共用的核心方法"""
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt}
    ]

    # 调用 DashScope API
    response = await self._call_dashscope(
        model=self.model,
        messages=messages,
        temperature=temperature or 0.7,
        max_tokens=max_tokens or 8000,
        response_format={"type": "json_object"} if json_mode else None
    )
    return response
```

**逐行讲解：**

- `system_prompt` — 告诉 LLM"你是谁、你要干什么"（角色设定）
- `user_prompt` — 告诉 LLM"具体任务是什么"（具体指令）
- `json_mode=True` — 要求 LLM 返回 JSON 格式，方便代码解析
- `temperature` — 控制随机性（0=确定，1=创造性）
- `max_tokens` — 限制输出长度，防止 token 消耗过大

**核心代码 — add_message：**

```python
def add_message(self, state, msg_type, data):
    """向 SSE 队列推送消息"""
    msg = {"type": msg_type, "agent": self.name, **data}
    state["messages"].append(msg)

    # 如果存在异步队列，推送到队列（用于实时流式输出）
    queue = state.get("_message_queue")
    if queue:
        try:
            asyncio.get_event_loop().call_soon_threadsafe(
                queue.put_nowait, msg
            )
        except:
            pass
```

**逐行讲解：**

- `state["messages"].append(msg)` — 所有消息存到 state，保留完整历史
- `queue.put_nowait(msg)` — 同时推送到异步队列，让 SSE 流立即发送给前端
- `call_soon_threadsafe` — 确保线程安全的队列操作

```python
async def _initial_planning(self, state):
    # 发送进度事件
    self.add_message(state, "research_step", {
        "step_type": "planning",
        "title": "研究计划",
        "status": "running"
    })

    # 调用 LLM 生成大纲（带重试）
    prompt = self.PLANNING_PROMPT.format(query=state["query"])
    for attempt in range(3):  # 最多重试 3 次
        response = await self.call_llm(
            system_prompt="你是一位专业的电商运营分析师...",
            user_prompt=prompt, json_mode=True
        )
        result = self.parse_json_response(response)
        if result and len(result.get("outline", [])) >= 3:
            break  # 成功

    # 更新状态
    state["outline"] = result["outline"]
    state["hypotheses"] = result.get("hypotheses", [])
    state["research_questions"] = result.get("research_questions", [])
    return state
```

**逐行讲解：**

- `add_message` — 推送进度事件，前端实时看到"规划中…"
- `format(query=state["query"])` — 把用户问题填入提示词模板
- 重试机制 — LLM 可能输出格式不对，最多重试 3 次
- `json_mode=True` — 要求 LLM 返回纯 JSON
- `parse_json_response` — 清理可能的 markdown 包装（去除 ```json ```）
- 判断标准：`len(outline) >= 3`，至少 3 个分析维度才算成功

```typescript
const sendChat = useCallback(async (target, message) => {
  target.loading = true

  // 发起深度研究请求
  const res = await api.session.deepsearch({
    query: message,
    session_id: id,
    search_modes: deviceState.searchModes,
    industry_id: industryState.currentIndustryId,  // 传行业ID
  })

  // 检查是否是流式响应
  if (!res.data || typeof res.data.getReader !== 'function') {
    target.content += '\n\n[错误：服务器返回了非流式响应]'
    return
  }

  // 读取 SSE 流
  const reader = res.data.getReader()
  const decoder = new TextDecoder('utf-8')

  while (true) {
    const { done, value } = await reader.read()
    if (done) break

    const text = decoder.decode(value)
    // 解析 data: {...}\n\n 格式的 SSE 事件
    const lines = text.split('\n')
    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const event = JSON.parse(line.slice(6))
        handleSSEEvent(event, target)  // 根据 type 更新 UI
      }
    }
  }
}, [id])
```

**逐行讲解：**

- `industry_id: industryState.currentIndustryId` — 前端选什么行业就传什么，后端据此切换数据
- `res.data.getReader()` — Fetch API 的流式读取器
- `TextDecoder('utf-8')` — 将字节流解码为文本
- `line.startsWith('data: ')` — SSE 协议解析

## 4.4 常见坑

### 坑 1：TypeScript 类型推断问题

`ResearchState` 是 TypedDict，访问嵌套字段时 IDE 可能提示类型错误。解决方案：使用 `state.get("key", default)` 而不是 `state["key"]`。

### 坑 2：Agent 的 asyncio 陷阱

Agent 的 `call_llm` 是 async，如果忘记 `await`，代码不会报错但 LLM 不会真正被调用，结果为空。

### 坑 3：ECharts 中文不显示

必须同时做两件事：① `registerLocale('ZH')` ② `init(el, null, { locale: 'ZH' })`。缺一不可。

### 坑 4：DataQueryer 行业切换不生效

行业 id 必须在请求链路中完整传递：`前端 select → deepsearch({industry_id}) → 后端 ResearchRequest → service.research() → graph.run() → create_initial_state() → DataQueryer.process()`。
